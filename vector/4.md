# Vector Databases

## 0. What Is A Vector Database?
**A database built to find **closest-in-meaning** vectors, not exact matches.
**
Normal DB: `WHERE email = 'x@y.com'` (exact match)
Vector DB: "find the 5 vectors closest to this query vector" (similarity search)

**Simple words:** normal DB finds exact matches. Vector DB finds "closest in meaning" matches.

**Interview answer:** "A vector database stores high-dimensional embeddings and is optimized for approximate nearest-neighbor similarity search, rather than exact-match lookups like a traditional relational database."

## What Is ANN (Approximate Nearest Neighbor)?
A search method that finds a **very good** match fast, instead of the **guaranteed best** match slowly — by narrowing down to a small likely-correct candidate set instead of checking every vector. Trades a little accuracy for a lot of speed. HNSW and IVF are both ANN techniques (different strategies, same goal).

## 1. Why Brute-Force Breaks At Scale → ANN Search
Brute-force = compare query against EVERY vector, one by one. Fine at small scale, gets slower as data grows.

Vector DBs exist to NOT break at scale — they use ANN search (see above) instead of checking everything.

**Practical result:** at 100k vectors, brute-force numpy (~22ms) was already slower than Chroma's ANN query (~7ms) — and the gap widens further as data grows (millions of vectors → brute-force = seconds, ANN = still milliseconds).

**Simple words:** brute-force search breaks at large scale. Vector DBs exist specifically to not break at large scale, using smart indexing tricks like HNSW.

**Interview answer:** "Brute-force nearest-neighbor search is O(n) per query, which becomes too slow at millions of vectors. Vector databases use ANN indexes like HNSW to avoid scanning everything, trading a small amount of recall for large speed gains."

## 2. HNSW (Hierarchical Navigable Small World)
A graph where each vector connects to a few "nearby" vectors. Search = hop node to node, always toward whatever neighbor is closest to the query, until no closer hop is found.

"Hierarchical" = multiple layers — sparse top layer for big fast jumps, denser lower layers for fine precision. Start at top, descend layer by layer.

**Recall vs speed:** fewer hops = faster but may miss the true best match (approximate, not exact) — this is why Chroma's top result differed slightly from brute-force's exact answer in our benchmark.

**Simple words:** like a multi-level map — bird's-eye view to reach the right neighborhood fast, then zoom in locally. Fast, but might not find the absolute closest match.

**Interview answer:** "HNSW is a graph-based ANN index connecting vectors to approximate neighbors across hierarchical layers — a sparse top layer for coarse navigation, dense lower layers for precision. Querying greedily hops toward closer neighbors layer by layer, trading exact recall for major speed gains."

## 3. IVF (Inverted File Index) — The Other ANN Family
Clusters vectors into groups first (like k-means), each with a center point (centroid). Query only compares against centroids first, then searches only within the closest cluster(s) — not the whole dataset.

Example: 1M vectors → 1,000 clusters (~1,000 each). Query matches cluster #42 → only search ~1,000 vectors instead of 1M.

HNSW = graph, hop toward answer. IVF = clusters, search nearest cluster only. Both are ANN. `pgvector` supports both.

**Simple words:** sort everyone into cities first — when someone asks "who's nearest to me," only search their city, not the whole country.

**Interview answer:** "IVF is a clustering-based ANN technique — vectors are grouped via k-means, and at query time only the nearest cluster(s) are searched instead of the full dataset. Different strategy from HNSW's graph traversal, but same goal: avoid brute-force scanning at scale. pgvector supports both."

## 4. Metadata Filtering + Vector Search
Vectors can carry extra structured fields (metadata), e.g. `{"year": 2024, "category": "legal"}`. Combine metadata `WHERE`-style filters with similarity search two ways:

- **Pre-filter**: narrow by metadata first, then similarity-search only within that subset. Guarantees correct filter, but small subsets can hurt ANN speed/quality.
- **Post-filter**: similarity-search everything first, then drop results that don't match metadata. Fast, but risk of 0 results if top matches don't pass the filter.

**Simple words:** pre-filter = narrow the room first, then find closest person in it. Post-filter = find closest people anywhere, then check if any are even in the right room — might find none.

**Interview answer:** "Pre-filtering narrows the search space before similarity search, guaranteeing filter compliance. Post-filtering runs similarity search first and filters after — faster, but risks too few/zero results if top-k matches don't satisfy the filter."

## 5. Index Build Parameters (ef_construction, m)
Knobs that control "how much effort to spend building/searching the graph." No need to memorize numbers — just know the trade-off.

- **m**: neighbor connections per node. Higher = better accuracy, more memory, slower build.
- **ef_construction**: how thoroughly the graph is built. Higher = better quality, slower build.
- **ef_search**: candidates explored at query time. Higher = better recall, slower query.

**Simple words:** more effort = more accurate but slower/costlier. Less effort = faster/cheaper but may miss the best match.

**Interview answer:** "HNSW exposes tunable params like m (graph connectivity) and ef_construction (build-time thoroughness) that trade off build time, memory, and recall. Tune based on latency/accuracy needs rather than memorizing values."
